## 项目背景

随着直播越来越普及，有时有一些突发奇想随时想要直播的需求，如朋友之间的直播聊天、临时直播会议、游戏过程快速开播、屏幕展示等等。本项目旨在轻量级的Web直播平台，支持主播Web开播，观众Web看直播。用户随时浏览器开播，亦可下载第三方推流软件进行推流直播。



## 项目目标

- 实现直播流在浏览器中播放
- 支持直播列表与直播间页面
- 实现基础互动功能(弹幕/聊天/连麦)
- 实现基本的用户登录注册功能

## 核心业务流程

### 观众观看流程

```
登录/进入首页
→ 浏览直播列表
→ 点击直播卡片
→ 进入直播间
→ 拉流播放
→ 观看 / 聊天 / 弹幕 / 连麦
```



### 主播开播流程

#### OBS开播：

```
点击“我要开播”
→ 选择“第三方推流（OBS）”
→ 查看推流地址与推流码
→ 在 OBS 中配置推流信息
→ 点击平台“开始直播”或直接推流
→ 推流成功
→ 直播状态切换为“直播中”
```

#### WebRTC开播（V1暂不做

**功能目标**：主播无需下载第三方推流软件，仅通过浏览器即可完成开播，适用于轻量化、随时直播的场景

**参与对象**

- 主播(浏览器端)
- 平台Web端
- 业务后端(直播会话/鉴权)
- 媒体服务器
- 观众浏览器

**前置条件**

- 主播已登录
- 主播已进入“我要开播界面”并点击选择WebRTC开播
- 主播浏览器支持WebRTC
- 主播已授权摄像头/麦克风/屏幕采集权限

**开播方式**

- 摄像头开播
- 屏幕共享开播
- 摄像头+屏幕共享

##### WebRTC开播业务流程

###### 进入开播页

主播点击首页右上角"我要开播"，选择"浏览器快速开播"

###### 填写开播信息

- 直播标题
- 分类
- 封面(可选)
- 开播类型(摄像头/屏幕共享)

###### 请求媒体权限

系统根据开播类型请求浏览器权限：

- 摄像头开播：请求摄像头+麦克风
- 屏幕共享开播：请求屏幕共享+可选麦克风

若用户拒绝则无法开播

###### 本地预览

成功获取媒体流后，前端展示本地预览，主播可确认：

- 画面是否正常
- 麦克风是否正常
- 是否切换设备
- 是否重新选择共享屏幕/窗口

###### 创建直播会话

主播点击"开始直播后"，前端调用后端接口创建本次直播会话

后端需完成：

- 创建直播记录
- 创建直播会话(LiveSession)
- 生成本次推流鉴权信息(PublicToken)
- 返回媒体服务器信令地址

###### 建立WebRTC推流连接

前端通过：

- getUserMedia()/getDisplayMedia()
- RTCPeerConnection
- WebSocket/信令服务

与媒体服务器建立连接，交换SDP/ICE信息，并将主播音视频信流推送到媒体服务器

###### 开播成功

媒体服务器确认推流成功后：

- 当前直播状态切换成"直播中(Live)"
- 首页展示该直播

###### 结束直播

主播点击"结束直播后":

- 前端关闭采集与WebRTC连接
- 后端结束直播会话
- 直播状态切换为"已结束(Ended)"
- 首页下架该直播

## 直播状态机

直播间与直播会话需要维护明确状态

### 状态定义

1. **Offline(未开播)**：主播当前未开播
2. **Preparing(准备中)**：主播已进入开播流程，但尚未成功推流
3. **Live(直播中)**：直播以成功开播，观众可正常观看
4. **Ended(已结束)**：本次直播已结束

### 状态切换触发条件

- 主播进入开播流程并创建直播会话：	Offline -> Preparing
- 媒体服务器确认推流成功：                       Preparing->Live
- 主播手动关播：                                          Live->Ended
- 推流中断超时：                                          Live->Ended





## 第三方推流设计

### 功能说明

平台为主播提供标准 RTMP 推流能力，兼容 OBS 等第三方推流工具

### 页面展示信息

主播可在“第三方推流”模式下查看：

- 推流服务器地址（RTMP Server）
- 推流码 / 推流凭证（Stream Key / Credential）

### 推流状态回调(Webhook)机制

1. 当 OBS 成功连接媒体服务器时，媒体服务器向业务后端发送 `on_publish` 异步回调事件，后端校验推流凭证，校验通过后将房间状态设为 `Live`。
2. 当 OBS 断开连接超过 30 秒（防网络抖动），媒体服务器向业务后端发送 `on_unpublish` 回调事件，后端将房间状态设为 `Ended`。

### 业务规则

- 每位主播拥有唯一推流凭证

- 推流凭证支持手动重置

- 推流时服务器需校验推流凭证合法性

- 非法推流应拒绝接入

- 主播成功推流后，系统切换为“直播中”



## 异常流程与边界场景

### WebRTC开播异常（V1暂不做

#### 1）用户拒绝媒体权限

- 提示“未授予媒体权限，无法开播”

#### 2）浏览器不支持

- 提示用户使用 Chrome / Edge 等受支持浏览器

#### 3）推流连接失败

- 提示“开播失败，请重试”
- 状态保持为 `Offline / Preparing`

#### 4）推流中断

- 若网络中断或页面关闭导致推流中断
- 系统自动结束直播会话

### 第三方推流异常

#### 1）推流凭证错误

- 媒体服务器拒绝推流
- 页面提示“推流失败，请检查推流信息”

#### 2）主播未开始推流

- 即使已打开 OBS，但未开始推流，直播状态不应切换为“直播中”

#### 3）OBS 停止推流

- 平台自动结束直播状态



## 非功能需求

### 性能要求

- 支持单场直播约 100 名观众同时观看（V1 目标）
- 直播间页面进入后应尽快开始播放（理想网络环境下 3 秒内）

### 延迟要求

- WebRTC 主播推流延迟目标：< 1 秒
- 观众播放延迟目标：1 ~ 5 秒（取决于播放协议）

### 兼容性要求

- 支持 Chrome / Edge 最新版本浏览器
- V1 暂不保证 Safari / Firefox 完整兼容

### 安全要求

- 仅登录主播可开播
- OBS 推流必须校验推流鉴权信息
- 非法推流应被拒绝
- 用户未登录不可发送互动消息
